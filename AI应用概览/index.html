    AI 应用概览

    学习路径 #AI应用

    关联：[[学习清单-AI应用与Agent]]

    如何区分常见应用形态

    问： 如何区分常见应用形态？

    可从三个维度区分：交互方式、是否用工具/外部能力、与业务系统的关系。

    按交互方式区分

    形态 典型交互 例子
    聊天机器人（Chatbot） 多轮对话，一问一答，偏开放话题 客服问答、闲聊、简单咨询
    智能助手（Assistant） 多轮对话，带明确任务（写邮件、总结、翻译） 个人助理、写作/翻译助手
    Copilot 在具体工具里「伴随操作」：补全、建议、解释当前内容 代码补全、文档写作建议、表格公式建议
    工作流自动化（Workflow / Agent） 用户给目标或触发条件，系统自动执行多步、可含工具调用 定时报告、自动订票、跨系统数据处理

    按是否用工具/外部能力区分

    • 纯对话：只靠 LLM + 上下文，不调 API、不查库。适合闲聊、简单写作、翻译、知识问答。
    • 带工具的对话：会调用搜索、计算器、数据库、API 等。适合要实时信息、精确计算、操作外部系统的场景。
    • 自动化流水线：事件/定时触发，按步骤执行（含 LLM 决策 + 工具调用）。适合后台任务、批处理、与现有系统集成。

    按与业务系统的关系区分

    • 独立应用：单独的产品/页面，用户主动来聊天或下指令。
    • 嵌入型（Copilot）：长在 IDE、Office、内部系统里，围绕当前文档/代码/数据给建议或执行操作。
    • 后台型（Agent/自动化）：用户设定目标或规则，系统在后台自动跑，结果通过通知/报表/工单反馈。

    一句话对照

    • 聊天机器人：多轮对话，偏开放、偏陪伴/咨询，可纯 LLM。
    • 智能助手：多轮对话 + 明确任务，常需要工具。
    • Copilot：嵌在具体工具里，对「当前内容」做补全、建议或解释。
    • 工作流/Agent 自动化：用户给目标或触发条件，系统自动多步执行（可含工具），不一定以对话为主。

    典型技术栈与代表产品

    问： 典型技术栈是什么，有什么代表产品？

    可按层次理解，每层有常见代表。

    1. 模型层（LLM）

    提供「大脑」：理解、生成、推理。

    类型 代表产品/服务
    闭源 API OpenAI（GPT-4o / GPT-4）、Anthropic（Claude）、Google（Gemini）
    国内商用 通义、文心、智谱、月之暗面等
    开源 Llama、Qwen、DeepSeek、Mistral 等

    2. 接入与编排层

    负责调用模型 API、组装提示、管理多轮与工具调用。

    类型 代表
    框架/SDK LangChain、LlamaIndex、Semantic Kernel、CrewAI、AutoGen
    直接调用 各厂商官方 SDK（openai、anthropic 等）

    3. 提示与上下文

    设计 system/user 提示、管理上下文窗口与 Token。多数在框架或自研代码中实现，无单独「产品」；可配合模板引擎或 PromptLayer 等做管理。

    4. 记忆与检索（RAG）

    长期记忆、知识库、检索增强。

    组件 代表
    向量库 Pinecone、Weaviate、Milvus、Chroma、pgvector
    嵌入模型 OpenAI Embeddings、Cohere、开源 sentence-transformers
    组装方式 多通过 LangChain / LlamaIndex 等与 LLM 串联

    5. 工具与外部能力

    搜索、计算、API、代码执行等。

    类型 代表
    搜索 Bing/Google API、Serper 等
    代码执行 E2B、Sandbox 等
    其他 自建或第三方业务 API

    6. 部署与运维

    托管、鉴权、限流、监控与评估。

    类型 代表
    托管/运行 Vercel、Cloudflare Workers、自建后端 + 模型 API
    可观测与评估 LangSmith、Langfuse 等

    简化栈示意

    前端 / 触发器 → 编排框架（LangChain 等）→ LLM API + 提示/上下文
    → 可选（向量库 + 嵌入、工具 API）→ 业务逻辑与存储

    「代表产品」往往是组合：例如 OpenAI API + LangChain + Pinecone 是一条常见技术栈；而 ChatGPT、GitHub Copilot、Notion AI 是这类栈封装后的终端产品形态。

    纯 LLM 对话 vs 需要工具/外部数据的边界

    问： 理解「纯 LLM 对话」与「需要工具/外部数据」的应用边界，该怎么理解？

    两种形态的定义

    • 纯 LLM 对话：只依赖「模型 + 你喂给它的文本（含多轮对话历史）」生成回复，不调用任何外部接口、不查库、不执行代码。输入输出都是文本，模型只做「根据已有上下文生成下一段话」。
    • 需要工具/外部数据：在对话中会调用搜索、计算器、数据库、API、代码执行等，把「外部结果」再塞回上下文，让模型基于这些结果继续推理或生成。

    边界怎么划（什么时候用哪种）

    用纯 LLM 就够的情况：

    • 不依赖「此刻真实世界」的信息（如今天天气、股价、新闻）。
    • 不需要精确计算、查表、查你的私有数据。
    • 不需要「做事」（发邮件、改数据库、调 API）。

    典型场景：闲聊、写作润色、翻译、用模型已有知识做概念解释、按规则生成格式（如大纲、模板）。

    必须上工具/外部数据的情况：

    • 要实时/准确事实：查天气、股价、最新文档、知识库。
    • 要精确计算或检索：数学、查表、在大量文档里找答案（RAG）。
    • 要产生副作用：发邮件、创建工单、改数据、调业务 API。

    典型场景：带搜索的问答、个人/企业知识库问答、订票/订餐、代码执行、报表生成（要跑真实数据）。

    一句话总结

    若任务只依赖「模型已经学到的知识 + 当前对话里已经出现的内容」，用纯 LLM；若依赖「实时/外部/私有数据」或「在系统里执行动作」，就要加工具或外部数据，这时往往就进入 Agent/带工具的形态。

    记忆与 RAG

    学习路径 #记忆 #RAG

    关联：[[学习清单-AI应用与Agent]]

    短期记忆和「对话历史」是一回事吗？上下文怎么管？

    问： 短期记忆是什么？和对话历史是一回事吗？上下文窗口怎么管、怎么截断？

    • 短期记忆：在当前会话里，Agent/模型能「记住」的内容，通常就是对话历史：用户说了啥、助手回了啥、中间若有工具调用则还有 tool 的输入输出。这些一起构成下一次推理的「上下文」。
    • 怎么管：模型有上下文长度上限（如 8K、32K token）。超出会报错或被平台截断。所以要做截断策略：例如只保留最近 N 轮对话；或按 token 数从新到旧保留，超出的部分丢弃或先摘要再保留一段「此前对话摘要」。这样既保留近期细节，又不会爆窗。

    长期记忆和 RAG 是什么关系？向量库、嵌入、检索干啥的？

    问： 长期记忆和 RAG 是什么关系？向量库、嵌入、检索流程分别干什么？

    • 长期记忆：希望模型能利用「当前对话之外」的信息，例如企业文档、知识库、历史工单。这些通常不全部塞进上下文，而是按需检索后再塞进去。RAG（检索增强生成） = 用检索到的内容增强当前 prompt，再让 LLM 生成回答，就是实现长期记忆的一种方式。
    • 向量库：把一段段文本变成向量（embedding）存起来；查询时把「用户问题」也变成向量，用相似度（如余弦）找出最相关的几段。向量库负责存、查这些向量。
    • 嵌入（embedding）：用嵌入模型把文本映射成固定长度的数字向量，语义相近的文本向量接近，便于「按意思找相似」。
    • 检索流程：用户问题 → 用嵌入模型得到 query 向量 → 在向量库里做相似度检索（如 Top-K）→ 取回 K 段原文 → 拼进 prompt 给 LLM。LLM 只看到「问题 + 检索到的片段」，据此生成答案。

    Top-K、重排序、和提示的拼接都是啥？怎么选、怎么拼？

    问： 检索策略：Top-K、重排序、与提示的拼接方式是什么？怎么选 K、怎么拼？

    • Top-K：向量检索时只取相似度最高的 K 条（如 K=5）。K 太小可能漏掉相关文档，K 太大则无关内容多、占 token、还干扰模型。通常从 3～10 试起，按效果调。
    • 重排序（rerank）：先用向量检索拉回较多候选（如 20 条），再用一个专门的重排序模型对「query + 每条候选」打分，取 Top-N（如 5）再拼进 prompt。重排序更准但更慢、更贵，适合对准确率要求高的场景。
    • 拼接方式：把检索到的段落按顺序拼成一段（或列表），放在 system 或 user 里，并明确标注「以下是参考内容，请仅基于此回答」之类，减少模型瞎编。若内容很长，可先做摘要再拼，或只拼与 query 最相关的几段。

    RAG 和 Agent 怎么一起用？何时检索、结果怎么给 LLM？

    问： 能将 RAG 与 Agent 结合吗？何时检索、如何把检索结果交给 LLM？

    • 何时检索：可以在用户发问后、第一次调 LLM 前先做一次检索，把结果塞进首轮 prompt；也可以让 Agent 自己决定——在提示里说明「若需要知识库再调用检索工具」，由模型在推理时发起「查知识库」的 tool call，再把检索结果当 observation 塞回去。前者简单固定，后者更灵活、但多一步决策。
    • 结果怎么给 LLM：检索结果当文本放进 messages（如 as tool 的 content，或 as user 里的一段「参考：…」）。在提示里写明「仅根据以下参考内容回答，若无法回答则说明依据不足」，有助于减少幻觉。
    • 结合方式：RAG 提供「长期记忆」的原料，Agent 负责在多步中决定何时查、查什么、以及结合工具结果与 RAG 结果一起推理。详见 [[Agent架构]]。

    多轮会话里记忆怎么持久化？怎么保证前后一致？

    问： 记忆的持久化与多轮会话中的一致性怎么理解？

    • 持久化：短期记忆（对话历史）若希望跨会话保留，需要你自己存：把每轮 user/assistant/tool 消息存库（按 session_id），下次进入同一会话时再加载进上下文。长期记忆（RAG）的向量库本身是持久化的，只要写入过，后续会话都可查。
    • 一致性：多轮里若上下文被截断或摘要，可能丢掉 earlier 的细节，模型会「忘记」前面说过的话，出现前后矛盾。缓解方式：截断时尽量保留与当前问题相关的轮次；或对较早对话做摘要，把「此前约定/结论」以摘要形式保留在上下文里，让模型始终「知道」之前的关键结论。

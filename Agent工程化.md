## LangChain、LlamaIndex、CrewAI、AutoGen 各是干啥的？该怎么选？

**问：** 主流框架的定位与选型：LangChain、LlamaIndex、CrewAI、AutoGen 各是干啥的？该怎么选？

- **LangChain**：偏**通用编排**。把 LLM 调用、提示模板、工具、链式步骤、记忆、RAG 都抽象成组件，用「链」或 LCEL 串起来。适合需要高度自定义、各种数据源和工具混用的 Agent 或流水线。生态大、文档多，但抽象多、学习成本高。
- **LlamaIndex**：偏**数据与检索**。擅长把各种数据源（文档、API、数据库）做成索引、做 RAG 查询，再和 LLM 结合。适合「以检索/知识库为核心」的应用；Agent 部分可自己写或配合 LangChain 等。
- **CrewAI**：偏**多 Agent 协作**。用 Role、Task、Crew 抽象，方便组一队 Agent 按顺序或按流程干活。适合明确「多人分工」的剧本式任务（如调研→写→审）。
- **AutoGen**：微软的**多 Agent 对话**框架。多个 Agent 可互相发消息、支持人机参与。适合对话式协作、辩论、需要人类中途插话的场景。

**怎么选**：若主要做 RAG + 简单 Agent，LlamaIndex + 少量自写逻辑可能就够。若单 Agent 多工具、链复杂，用 LangChain。若明确多角色协作剧本，用 CrewAI 或 AutoGen。也可以混用（如 LlamaIndex 做检索，LangChain 做 Agent 编排）。

## 评估 Agent 要看哪些维度？幻觉、延迟、成本怎么衡量？

**问：** 评估维度：幻觉、延迟、成本、稳定性与可观测性怎么掌握？

- **幻觉**：模型生成与事实或给定上下文不符的内容。可人工抽样标注「是否幻觉」；或用规则/模型做一致性检查（如 RAG 场景：答案中的关键事实是否能在检索结果中找到）。减少幻觉靠：更好的提示、RAG 约束、输出校验、或 Reflection。
- **延迟**：从用户发起到拿到最终回复的时间。可看 P50/P99、以及「首 token 时间」（流式时的体感）。多步 Agent 要加总每步 LLM + 工具调用时间；优化方向包括换更快模型、减上下文、并行工具调用、缓存。
- **成本**：按 token 计费时，统计每请求/每用户的 prompt_tokens 与 completion_tokens，乘单价。可对单用户/单日设上限并告警。成本优化：模型选型、压缩提示、截断历史、缓存重复前缀。
- **稳定性与可观测性**：成功率、错误类型（超时、限流、格式错误）；关键步骤打日志、上报 trace（如 LangSmith、Langfuse），便于排查「哪一步挂了、模型返回了啥」。

## 日志、追踪、监控在调试和运维里具体怎么用？

**问：** 日志、追踪与监控在 Agent 调试与运维中的作用是什么？

- **日志**：每轮请求记录「输入消息摘要、调用了哪些工具、工具返回摘要、最终输出、token 消耗、错误信息」。便于事后排查「用户说了啥、模型选了啥工具、结果对不对」。
- **追踪（Trace）**：把一次用户请求拆成**多步 span**（如：LLM 调用 1、tool A、LLM 调用 2、tool B、LLM 调用 3），记录每步的输入输出和耗时。这样能一眼看出延迟和成本出在哪一步，方便优化和排错。
- **监控**：对延迟、错误率、token 消耗、成本做**聚合与告警**（如 P99 延迟 > 10s 告警、单日成本超阈值告警）。调试时靠日志和 trace；上线后靠监控发现异常再下钻到具体请求的 trace。

## 安全与合规要考虑哪些？输入输出过滤、权限、隐私怎么做？

**问：** 安全与合规：输入输出过滤、权限、数据隐私与审计要考虑什么？

- **输入过滤**：对用户输入做长度限制、敏感词/违禁词过滤、以及简单的注入检测（如是否包含「忽略以上指令」类模式），减少 Prompt Injection 和滥用。
- **输出过滤**：对模型输出做脱敏（如手机号、身份证打码）、禁止输出内部 prompt 或系统信息；若输出会直接展示给用户，可再做一次策略检查。
- **权限**：工具与数据按「谁能用」做最小权限。例如删数据、发邮件只允许授权 Agent 或需二次确认；RAG 检索可按用户/租户做数据隔离。
- **数据隐私与审计**：明确哪些数据会进 LLM（含第三方 API），是否符合隐私政策；对敏感操作留审计日志（谁、何时、调了啥、结果摘要），便于合规与事后调查。

## 部署形态有哪些？和现有系统怎么集成？

**问：** 常见部署形态：API 服务、异步任务、与现有系统集成怎么做？

- **API 服务**：把 Agent 包成 HTTP/gRPC 接口，前端或其它服务调。同步时「请求→多轮推理与工具→返回最终回复」；若超时风险大，可做成「提交任务 → 轮询/回调拿结果」的异步接口。
- **异步任务**：用户请求进队列，Worker 消费后跑 Agent，结果写库或发通知。适合允许延迟的场景（如报告生成、批量处理），可复用现有任务队列（Celery、SQS 等）。
- **与现有系统集成**：Agent 作为内部服务，被现有业务逻辑调用（如客服系统里「智能回复」按钮调 Agent）；或 Agent 通过工具调用现有系统的 API、数据库。集成时注意鉴权、限流、错误码与重试策略一致。

    多 Agent 系统

    学习路径 #agent #多Agent

    关联：[[学习清单-AI应用与Agent]]

    多 Agent 典型用在什么场景？角色怎么划分？

    问： 多 Agent 的典型场景与角色划分是什么？比如研究员、写代码、审核怎么分？

    • 典型场景：任务复杂、且天然可拆成多种能力时。例如：写一份行业报告 = 需要有人查资料、有人写稿、有人审稿；做一个功能 = 有人设计、有人写代码、有人测。单 Agent 也可以靠多工具完成，但多 Agent 把「谁负责哪一步」显式化成不同角色，便于分工、约束和迭代。
    • 角色划分：按职责拆，每个 Agent 有固定角色（system 提示里写清）。例如：研究员只做检索与摘要，不写结论；写手根据研究员给的素材写正文；审核只做事实与逻辑检查、提出修改意见。这样每个 Agent 的 prompt 简单、行为更可控，且人类可以针对某个角色调优或替换模型。

    顺序、并行、辩论、路由这些协调方式分别是什么意思？

    问： 多 Agent 的协调方式：顺序执行、并行、辩论、路由/调度分别是什么意思？

    • 顺序执行：Agent A 做完把结果给 B，B 做完给 C……像流水线。适合步骤强依赖（先调研再写再审）。
    • 并行：多个 Agent 同时跑，各自处理不同子任务（如同时查三份资料），最后把结果汇总。适合可独立拆分的子任务，能省时间。
    • 辩论：多个 Agent 对同一问题各自给出答案或立场，再互相批评、补充，最后由一个 Agent 或人类综合结论。适合需要多角度、减少单一模型偏见或错误的场景。
    • 路由/调度：由一个协调者（可以是规则或一个小模型）根据用户意图或当前状态，决定「下一步叫哪个 Agent、传什么输入」。适合任务分支多、路径不固定的情况。

    多 Agent 之间传什么格式的消息？状态怎么共享？

    问： 多 Agent 间的消息格式与状态共享怎么做？

    • 消息格式：通常用统一结构，例如每条消息包含：发送者（agent_id）、角色、内容（文本或结构化）、可选 metadata（时间、任务 id）。内容可以是自然语言，也可以是「任务描述 + 输入数据」的 JSON，便于下一个 Agent 解析。和单 Agent 的 user/assistant/tool 类似，只是「发送者」变成不同 Agent。
    • 状态共享：可以有一个共享上下文（如一份共享的 dict 或文档），每个 Agent 读当前状态、写自己的输出进去，下一个 Agent 只读自己需要的部分。或者用消息队列：每个 Agent 只消费上家的输出、产出给下家，状态完全通过消息传递，无共享内存。选哪种取决于是否需要多 Agent 反复读同一份中间结果。

    什么时候值得上多 Agent，而不是一个 Agent 多几个工具？

    问： 何时用多 Agent 而非单 Agent 加更多工具？

    • 单 Agent 多工具：一个模型、多工具，模型自己决定何时用哪个工具。适合步骤不太多、决策路径相对清晰的任务，实现简单、延迟低。
    • 多 Agent：适合步骤多、且每步需要不同「人设」或强约束时。例如必须「先调研再写、写的人不能自己查」，用两个 Agent 更容易约束；或需要「多人辩论再综合」，单 Agent 难以模拟多立场。若只是「一个助手会查会算会写」，单 Agent + 多工具通常就够。多 Agent 会带来更多调用、更多上下文传递和协调逻辑，所以只在「分工带来明显收益」时用。

    Crew、Team、Role 这些框架里的概念大概指什么？

    问： 主流多 Agent 框架的抽象（Crew、Team、Role）是什么？

    • Role（角色）：对应一个 Agent 的身份与职责，通常包含 name、goal、backstory 或 system prompt，以及它能用的工具。例如 "研究员" role：目标是收集并摘要信息，只能用搜索与阅读工具。
    • Team / Crew：一组 Agent 的集合，并定义他们之间的协作方式。例如 Crew：一组 Role，按顺序或按图执行任务，由一个「主管」分配任务或收结果。Team 类似，有的框架里指「共享目标的一队 Agent」。
    • 不同框架（CrewAI、AutoGen 等）用词略有差异，但核心都是：Role = 谁做什么，Crew/Team = 谁和谁怎么配合；实现上多为「每个 Role 一个 LLM 调用配置 + 工具集，Crew 负责调度与传消息」。
